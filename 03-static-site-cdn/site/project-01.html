Why this architecture?

This service exposes a small REST API with a serverless backend to minimize operational overhead. The design prioritizes least-privilege access, predictable scaling, and low idle cost.

Architecture

Client → API Gateway → Lambda → DynamoDB

Scaling model
Automatic scaling with request volume
Cost model
Pay-per-request, $0 idle cost
Persistence
Key-value access via taskId

Operational visibility

Lambda writes invocation logs to CloudWatch Logs. Errors and latency can be viewed via Lambda metrics.

Key trade-offs
Pros
No servers, automatic scaling, low cost
Cons
Cold starts, stateless constraints, service limits
Decisions documented: serverless compute for zero idle cost; DynamoDB for simple access patterns; least-privilege IAM; CloudWatch for visibility.
