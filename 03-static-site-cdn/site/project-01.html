Why this architecture?

The goal was a REST API with minimal operations overhead and near-zero idle cost. Lambda provides event-driven compute, DynamoDB provides managed persistence, and API Gateway exposes a clean HTTP interface.

Architecture

Client → API Gateway → Lambda → DynamoDB

Scaling model
Automatic scaling with request volume
Cost model
Pay-per-request, $0 idle cost
Persistence
Key-value access via taskId

Operational visibility

Lambda writes invocation logs to CloudWatch Logs. Errors and latency can be viewed via Lambda metrics.

Key trade-offs
Pros
No servers, automatic scaling, low cost
Cons
Cold starts, stateless constraints, service limits
Tip: In interviews, explain the trade-off (Lambda vs EC2, DynamoDB vs RDS) before the implementation details.
