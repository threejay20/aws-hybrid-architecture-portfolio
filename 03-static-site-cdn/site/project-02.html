Why this architecture?

The goal was to decouple producers and consumers and handle failures safely. SNS provides fanout, SQS provides buffering and retries, Lambda processes messages, and a DLQ isolates poison-pill messages.

Architecture

Producer → SNS → SQS → Lambda (with DLQ for failures)

Decoupling
Producer is isolated from consumer speed/failures
Reliability
Retries + DLQ prevent message loss
Scaling model
Lambda scales with queue depth

Failure handling (DLQ)

Messages are retried automatically. If processing keeps failing and the message exceeds the max receive count, SQS moves it to the DLQ for inspection and reprocessing.

Key concepts: - Visibility timeout - At-least-once delivery (duplicates possible) - maxReceiveCount → DLQ routing
Troubleshooting workflow
If errors spike
Check CloudWatch Logs for Lambda failures
If queue backs up
Check Lambda concurrency + message age
If DLQ grows
Inspect message body; fix code/data; redrive
Tip: If the exam mentions “poison pill” or “don’t lose messages,” think SQS retries + DLQ.
